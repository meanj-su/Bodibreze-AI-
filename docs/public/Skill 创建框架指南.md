# Skill 创建框架指南

本指南用于设计、创建、维护和迭代 Codex Skill。一个好的 Skill 不是面向人的长篇说明书，而是写给另一个 AI Agent 的标准作业流程 SOP：明确什么请求会触发、执行前需要什么输入、按什么顺序操作、遇到异常如何分支、使用哪些脚本和资料、如何验证完成，以及哪些操作不能擅自执行。

## 1. 核心原则

- Skill 的核心目标是让 Agent 稳定执行任务，而不是堆放背景知识。
- `description` 是主要触发入口，必须写清能力、触发场景、相关平台、文件类型和相似请求。
- 主流程只保留必要步骤；详细业务规则、字段口径、排错资料放入 `references/`。
- 稳定、重复、易错的步骤沉淀为 `scripts/`，不要让 Agent 每次重写。
- 原始数据、生产文件、下载文件默认 0 修改；清洗、筛选、公式和格式处理只能发生在派生文件、临时输出或写入层。
- 已验证流程优先复用；只有异常或验证失败时才进入深度 debug。
- 不影响主流程的 debug 延后处理，避免把临时排查写成全局规则。
- 涉及生产写入、发布、删除、外部通知、账号 Token 或敏感凭证时，必须明确授权边界。

## 2. 推荐目录结构

```text
skill-name/
├── SKILL.md                 # 必需：入口、流程、规则、边界
├── agents/
│   └── openai.yaml          # 推荐：Skill 展示名称、简短描述、默认提示
├── scripts/                 # 可选：稳定、重复执行的脚本
├── references/              # 可选：详细业务规则、字段口径、API 文档、排错说明
└── assets/                  # 可选：模板、图片、字体、示例资源
```

不要额外创建无必要的 `README.md`、`CHANGELOG.md`、`QUICK_REFERENCE.md`、`INSTALLATION_GUIDE.md` 等文件。若信息需要跨项目复用，优先写入本公共 docs；若只服务某个 Skill，放到该 Skill 的 `references/`。

## 3. SKILL.md 标准结构

后续所有 Skill 优先采用以下结构：

```text
YAML frontmatter
Objective
Required Inputs
Quick Commands
Workflow
Decision Rules
Module Rules
Resources
Tool Usage
Output Requirements
Validation
Safety Constraints
Completion Criteria
```

各部分职责：

| 部分 | 必须回答的问题 |
| --- | --- |
| YAML `name` | Skill 的唯一名称是什么 |
| YAML `description` | 什么请求、平台、文件或场景会触发该 Skill |
| Objective | 最终要完成什么结果 |
| Required Inputs | 执行前需要哪些输入；缺失时哪些可默认、哪些必须确认 |
| Quick Commands | 用户或 Agent 最短可用启动命令 |
| Workflow | 标准执行顺序是什么 |
| Decision Rules | 异常、分支、复用、跳过和停止条件是什么 |
| Module Rules | 多国家、多板块、多平台等模块差异如何处理 |
| Resources | 什么情况下读取哪个脚本、参考文件或模板 |
| Tool Usage | CLI、API、MCP、浏览器、表格工具如何使用 |
| Output Requirements | 最终输出必须包含什么 |
| Validation | 完成前必须检查什么 |
| Safety Constraints | 哪些操作禁止或需要先确认 |
| Completion Criteria | 满足什么条件才算真正完成 |

## 4. YAML frontmatter 规则

`SKILL.md` 的 YAML frontmatter 只保留：

```yaml
---
name: skill-name
description: 说明 Skill 能做什么，以及什么用户请求、平台、文件、任务场景应该触发它。Use when ...
---
```

规则：

- `name` 只使用小写英文字母、数字和连字符。
- 文件夹名必须与 `name` 一致。
- 名称控制在 64 个字符以内，优先短、清晰、动词导向。
- `description` 同时写清能力和触发条件，不要只写“用于分析数据”。
- “什么时候使用该 Skill”的信息必须写进 `description`，不要只写在正文里。
- 不添加无法被 Skill 机制识别的自定义 frontmatter 字段。

## 5. 自由度控制

根据任务风险决定 Skill 写多细：

- 高自由度：适合创意、总结、策略建议。写目标、约束和判断原则，允许 Agent 按上下文发挥。
- 中自由度：适合数据分析、报告生成。固定核心指标、口径和输出结构，允许按异常扩展分析。
- 低自由度：适合生产写入、下载链路、部署、财务数据、批量文件处理。明确命令、顺序、停止条件和验证点。

判断标准：任务越容易造成数据错误、外部影响或生产事故，流程越要具体；任一步失败都不得假装完成。

## 6. Resources 拆分规则

放入 `references/`：

- 数据表结构、字段定义、指标口径。
- API 文档、平台规则、业务规则。
- 多国家、多品牌、多板块的差异说明。
- 详细故障排查手册和大量示例。
- 超过 100 行的参考文件建议在顶部加目录。

放入 `scripts/`：

- 每次都会重复执行的数据处理逻辑。
- Excel/CSV 导入导出、格式转换、数据质量验证。
- API 拉取、浏览器自动化、固定部署或检查流程。
- 新增脚本后必须实际运行或至少做语法检查，并在最终反馈中说明验证边界。

放入 `assets/`：

- Excel、PPT、Word、HTML、React 等模板。
- 品牌 Logo、图片、字体、示例配置。
- 用于最终产物生成的静态资源。

`SKILL.md` 只写资源导航，不一次性塞入所有详细资料；所有参考文件应直接从 `SKILL.md` 可发现，避免多层嵌套引用。

## 7. 标准开发流程

1. 收集真实用户请求示例，明确应该触发和不应该触发的场景。
2. 确认必需输入、默认值、缺失信息时的确认规则。
3. 判断哪些流程应写入 `SKILL.md`，哪些应拆到 `references/`、`scripts/`、`assets/`。
4. 初始化或整理 Skill 目录。
5. 编写或更新资源文件。
6. 编写 `SKILL.md`，保持 SOP 化和命令式表达。
7. 更新 `agents/openai.yaml`，确保展示名称、简短描述和默认提示与 `SKILL.md` 一致。
8. 运行格式、语法或项目检查。
9. 用真实任务或代表性任务测试。
10. 根据失败点迭代 Skill，但不要把临时排查细节污染主流程。

## 8. 维护迭代规则

- 新增国家、平台、URL、profile、模块、输出路径、时间节奏时，优先改配置，不硬编码。
- 新增模块前，先判断是否能复用已有流程：页面定位、参数选择、导出、下载、合并、写入、验证。
- 发现重复 debug、重复点击、重复页面检查时，先把已验证步骤抽成可复用函数或 SOP 分支。
- 对下载、写表、部署等低自由度任务，失败要输出具体原因和下一步动作。
- 对已成功的第一天/首站点流程，后续同类型任务优先跟随前序操作；异常时再检查页面和资源。
- 不要把“命令已运行”当成“任务已完成”；必须检查目标文件、数据范围、写入结果或可验证状态。
- 原始文件永远保持 0 修改；任何清洗、筛选、排序、公式处理都在派生文件或写入层完成。

## 9. 输出要求

完成 Skill 开发或维护后，最终反馈至少包含：

- 修改了哪些 Skill、脚本、配置或公共 docs。
- 验证命令和验证结果。
- 哪些行为已真实测试，哪些只是语法检查或实现完成。
- 失败或未覆盖项的具体原因。
- 后续使用的最短命令或触发方式。

执行真实业务任务后，最终反馈至少包含：

- 成功/失败任务列表。
- 原始文件路径和派生文件路径。
- 在线表或外部系统同步状态。
- 数据范围、关键筛选条件和验证结果。
- 各流程耗时。
- 需要用户处理的事项。

## 10. 发布前检查清单

- `name` 只包含小写字母、数字和连字符。
- Skill 文件夹名与 `name` 一致。
- `description` 同时包含能力说明和触发场景。
- YAML frontmatter 只包含 `name` 和 `description`。
- 核心步骤使用命令式表达。
- Required Inputs、Workflow、Decision Rules、Validation、Completion Criteria 明确。
- 说明了哪些情况需要用户确认。
- 详细资料已移到 `references/`，重复逻辑已移到 `scripts/`。
- 没有重复维护同一规则。
- 没有密钥、Token、账号或其他敏感信息。
- 新增脚本已经实际运行测试，或明确说明未能测试的原因。
- 至少用一个真实或代表性任务验证触发和执行效果。

## 11. 项目级落地要求

- 所有 `E:\AI项目汇总` 下项目在新增或维护 Skill 前，优先读取本指南。
- 项目内已有 Skill 与本指南冲突时，以本指南为优先级更高的公共规范；项目特殊规则只作为局部补充。
- 跨项目可复用的 Skill 规则、字段口径、输出规范优先沉淀到 `E:\AI项目汇总\公共docs`。
- 不要在各项目复制本指南；项目目录通过 `公共docs` 联接读取同一份规范。
- 对 Shopee、AdsPower、Google Sheets、报表生成、部署发布等高复用流程，优先沉淀为脚本和 SOP，不依赖聊天记录。
